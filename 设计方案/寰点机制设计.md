# 寰点机制设计

## 1. 设计目标

寰域的核心不是传统论坛中的“分类字段”，而是以**寰点**作为最小可治理语义单元，以**向量**作为底层关系表达方式，共同完成内容组织、用户兴趣建模、推荐分发、域聚合。

本设计文档约束第一版寰点系统的落地方式，目标如下：

向量化的关系表达、召回与排序细节，配套参考《混合语义模型设计》。

1. 所有寰点统一进入一个寰点池。
2. 允许近义词、同义词、缩写、全称在早期共存。
3. 用户在发布、浏览、点赞、收藏、屏蔽内容时，与寰点产生显式关系并形成粘性。
4. 系统通过向量化表达文章、用户、寰点之间的隐式语义关系。
5. 系统通过“使用表现 + 认可度 + 内容反馈 + 语义相似度”判断寰点是否合理。
6. 对低质量寰点执行回收，对高度重复寰点执行归并。

## 2. 核心概念

### 2.1 寰点

寰点不是全部语义本身，而是社区中可命名、可投票、可治理、可聚合的显式语义单元，可以是：

- 开发语言，如 `Go`、`Python`、`TypeScript`
- 技术框架，如 `Spring Boot`、`React`、`Vue`
- 应用场景，如 `爬虫`、`数据分析`、`AI Agent`
- 技术能力，如 `性能优化`、`微服务`、`DevOps`

### 2.2 寰点池

全站所有寰点统一进入一个总寰点池，不按域拆分独立池。

设计原则：

- 新寰点允许产生。
- 同义词和近义词允许暂时共存。
- 是否保留、合并、回收，由系统和用户共同决定。

### 2.3 寰点粘性

寰点粘性用于描述用户和某个寰点之间的兴趣强度。

典型来源：

- 用户发布带某寰点的文章
- 用户阅读带某寰点的文章
- 用户点赞、收藏、关注某寰点
- 用户对某寰点选择“不感兴趣”或“屏蔽”

这里的粘性主要表达**显式关系**，即用户已经通过清晰行为表达出来的兴趣强弱。

### 2.4 主寰点与别名寰点

由于第一版允许同义词共存，因此系统内部保留两类身份：

- 主寰点：作为推荐、聚合、统计的标准承载对象
- 别名寰点：用户真实输入的寰点，允许映射到某个主寰点

示例：

- `JavaScript` 是主寰点
- `JS`、`Javascript` 可以是别名寰点

第一版不强制新建寰点必须选择主寰点，但允许系统在后续治理阶段把别名寰点归并到主寰点。

### 2.5 向量关系

为了避免“寰点颗粒不足以完整表达文章语义”的问题，系统同时维护三类向量：

- 文章向量：表达一篇文章的整体语义
- 寰点向量：表达一个寰点在语义空间中的中心位置
- 用户向量：表达一个用户长期与短期兴趣的综合位置

向量承担的是**隐式关系表达**，典型用途如下：

- 文章向量与寰点向量相似，用于推荐候选寰点
- 用户向量与寰点向量相似，用于发现潜在兴趣
- 用户向量与文章向量相似，用于推荐内容
- 寰点向量与寰点向量相似，用于辅助归并和关系发现

### 2.6 双通道关系

寰域中的核心关系不再只是一条“粘性分”，而是两条并行通道：

- 显式通道：通过发布、选择、点赞、收藏、屏蔽、投票形成可解释关系
- 隐式通道：通过向量相似度表达尚未被用户明确说出的语义接近度

因此，用户与寰点、文章与寰点、用户与文章之间都应同时保留：

- 显式关系
- 隐式关系
- 最终融合分

## 3. 生命周期设计

### 3.1 状态定义

| 状态 | 含义 | 前台展示 | 是否参与推荐 | 是否保留历史数据 |
| --- | --- | --- | --- | --- |
| `observing` | 新建寰点，处于观察期 | 是 | 低权重或不参与 | 是 |
| `active` | 正式寰点，已验证合理 | 是 | 是 | 是 |
| `pending_merge` | 疑似同义词或重复寰点 | 是 | 降权 | 是 |
| `merged` | 已归并到主寰点 | 可保留原展示 | 通过主寰点参与 | 是 |
| `low_activity` | 合理但近期活跃度低 | 是 | 降权 | 是 |
| `pending_recycle` | 已满足低质量回收条件 | 弱展示 | 否 | 是 |
| `recycled` | 已回收，不再作为有效寰点使用 | 否 | 否 | 是 |

### 3.2 生命周期说明

#### `observing`

新寰点首次进入寰点池时的默认状态。

系统策略：

- 可以挂载到文章详情页
- 可以被搜索到
- 不进入热门寰点池，或权重很低
- 开始累计使用、投票、内容表现数据

#### `active`

已被证明是合理且具有持续使用价值的正式寰点。

系统策略：

- 可进入推荐、聚合、筛选
- 可出现在热门寰点、域寰点池、搜索建议中
- 参与用户画像和内容推荐计算

#### `pending_merge`

寰点本身不是垃圾，但高度疑似另一个寰点的同义词、缩写或拼写变体。

系统策略：

- 前台继续展示，避免用户认知断层
- 后台进入待合并列表
- 降低该寰点自身的推荐权重

#### `merged`

寰点已确认归并到某个主寰点。

系统策略：

- 原寰点不再作为独立推荐对象
- 搜索命中时跳转到主寰点聚合页
- 历史文章仍保留原始寰点记录

#### `low_activity`

寰点本身合理，但最近一段时间使用变少。

系统策略：

- 不回收历史内容
- 不出现在热门池
- 暂时保留搜索和详情访问能力

#### `pending_recycle`

寰点已经具备较充分样本，同时表现长期偏弱，进入回收观察名单。

系统策略：

- 不再进入推荐
- 不再作为建寰点建议优先项
- 继续观察一段时间，等待恢复或正式回收

#### `recycled`

寰点被系统正式回收，但只做逻辑回收，不做物理删除。

系统策略：

- 不参与推荐、聚合、热榜
- 不在寰点池中展示
- 保留历史数据、投票、文章挂载关系

## 4. 生命周期流转规则

### 4.1 基本流转路径

```text
新建 -> observing
observing -> active
observing -> pending_merge
observing -> pending_recycle
active -> low_activity
active -> pending_merge
low_activity -> active
low_activity -> pending_recycle
pending_merge -> merged
pending_recycle -> recycled
recycled -> observing
```

### 4.2 状态流转阈值

#### `observing -> active`

满足以下任一条件即可：

1. 近 30 天内被至少 10 篇文章使用，且至少 5 个不同作者使用
2. 总投票数至少 20，赞同率至少 70%
3. `QualityScore >= 65`

#### `observing -> pending_merge`

满足以下全部条件：

1. `MergeScore >= 80`
2. 已存在候选主寰点
3. 候选主寰点处于 `active` 状态

#### `observing -> pending_recycle`

满足以下全部条件：

1. 已观察至少 14 天
2. 近 30 天使用文章数少于 5
3. 近 30 天不同作者数少于 3
4. 总投票数至少 10，赞同率低于 30%
5. `QualityScore < 30`

#### `active -> low_activity`

满足以下全部条件：

1. 连续 30 天活跃度偏低
2. 近 30 天文章使用数少于 3
3. 近 30 天无明显关注、收藏增长

#### `low_activity -> active`

满足以下任一条件：

1. 近 30 天重新达到正式寰点阈值
2. 出现新的高质量文章持续使用该寰点

#### `active / low_activity -> pending_recycle`

满足以下全部条件：

1. 连续 90 天低活跃
2. 内容表现长期偏低
3. 赞同率长期偏低，且样本足够
4. 无法被合理归并到已有主寰点

#### `pending_merge -> merged`

满足以下任一条件：

1. 系统判定置信度高，自动归并
2. 管理员确认归并

#### `pending_recycle -> recycled`

满足以下全部条件：

1. 再观察 14 天仍无改善
2. 无管理员人工保留
3. 无核心专题或重要内容依赖

#### `recycled -> observing`

满足以下任一条件：

1. 重新被较多用户主动使用
2. 管理员人工恢复

## 5. 评分规则

第一版不建议只靠“赞同 / 不赞同”判断寰点是否保留，而是引入四类指标共同决定：

1. 使用分 `UsageScore`
2. 认可分 `ApprovalScore`
3. 内容表现分 `ContentScore`
4. 合并分 `MergeScore`

其中前三项决定寰点质量，第四项决定是否应归并。向量相似度既参与 `MergeScore`，也可作为 `ContentScore` 和推荐排序的辅助信号。

### 5.1 使用分 `UsageScore`

用于衡量寰点是否被真实使用。

统计维度：

- `post_count_30d`：近 30 天被多少篇文章使用
- `unique_authors_30d`：近 30 天有多少不同作者使用
- `active_days_30d`：近 30 天分布在多少个自然日出现

建议权重：

- 文章数占 50%
- 作者数占 30%
- 活跃天数占 20%

示意公式：

```text
UsageScore =
0.5 * normalize(post_count_30d) +
0.3 * normalize(unique_authors_30d) +
0.2 * normalize(active_days_30d)
```

### 5.2 认可分 `ApprovalScore`

用于衡量社区是否认可该寰点命名合理、语义准确、使用得当。

投票规则：

- 赞同：`approve`
- 不赞同：`disapprove`

统计维度：

- `approve_count`
- `disapprove_count`
- `vote_count = approve_count + disapprove_count`
- `approve_rate = approve_count / vote_count`

第一版建议加入样本保护，防止少量投票导致误杀：

- `vote_count < 10`：只做参考，不参与回收
- `10 <= vote_count < 20`：弱参与质量判断
- `vote_count >= 20`：作为正式判断依据

示意公式：

```text
ApprovalScore = approve_rate * min(1, vote_count / 20) * 100
```

### 5.3 内容表现分 `ContentScore`

用于衡量带该寰点的内容，是否被用户真实消费和认可。

统计维度：

- `view_count_30d`
- `read_complete_rate_30d`
- `like_rate_30d`
- `favorite_rate_30d`
- `negative_feedback_rate_30d`

建议权重：

- 阅读完成率占 35%
- 点赞率占 20%
- 收藏率占 25%
- 负反馈率占 -20%

示意公式：

```text
ContentScore =
0.35 * normalize(read_complete_rate_30d) +
0.20 * normalize(like_rate_30d) +
0.25 * normalize(favorite_rate_30d) -
0.20 * normalize(negative_feedback_rate_30d)
```

### 5.4 合并分 `MergeScore`

用于衡量该寰点是否与某个已有寰点高度重复。

统计维度：

- 名称相似度
- 文章重合度
- 用户建议合并次数
- 语义相似度

建议权重：

- 名称相似度占 30%
- 文章重合度占 30%
- 合并建议占 20%
- 语义相似度占 20%

示意公式：

```text
MergeScore =
0.30 * NameSimilarity +
0.30 * PostOverlap +
0.20 * MergeVotes +
0.20 * SemanticSimilarity
```

### 5.5 质量分 `QualityScore`

用于综合判断寰点是否值得保留。

建议公式：

```text
QualityScore =
0.35 * UsageScore +
0.30 * ApprovalScore +
0.20 * ContentScore +
0.15 * StabilityScore
```

其中 `StabilityScore` 用于衡量寰点是否持续出现，而不是短期异常波动。

## 6. 治理保护规则

### 6.1 小样本保护

- 样本不足时不做回收判断
- 新建寰点至少有 14 天观察期
- 投票数不足 10 的寰点，不得仅凭认可度进入回收

### 6.2 新技术保护

对于新兴技术或短期爆发型概念，可延长观察期，例如：

- `MCP`
- `Bun`
- `RAG`
- `AI Agent`

这类寰点即使样本小，也不能被快速回收。

### 6.3 回收不是删除

寰点回收只做逻辑回收，不做物理删除。

必须保留：

- 历史文章挂载关系
- 历史投票记录
- 历史粘性数据
- 历史归并关系

### 6.4 人工兜底

后台必须提供人工干预能力：

- 强制保留寰点
- 强制回收寰点
- 指定主寰点
- 手动执行归并
- 恢复已回收寰点

## 7. 用户行为、寰点粘性与向量关系

用户行为既影响用户画像，也影响寰点评分。

### 7.1 粘性行为建议

| 行为 | 对用户-寰点粘性的影响 |
| --- | --- |
| 发布带寰点的文章 | `+3` |
| 手动确认 AI 推荐的寰点 | `+2` |
| 阅读文章 | `+1` |
| 阅读达到阈值 | `+1` |
| 点赞 | `+2` |
| 收藏 | `+3` |
| 关注寰点 | `+4` |
| 不感兴趣 | `-3` |
| 屏蔽寰点 | `-8` |

### 7.2 对寰点评分的反哺

用户行为除了影响用户画像，还可反哺寰点系统：

- 更多文章使用，提升 `UsageScore`
- 更多赞同投票，提升 `ApprovalScore`
- 更高阅读完成率和收藏率，提升 `ContentScore`
- 更多合并建议，提升 `MergeScore`

### 7.3 显式关系与隐式关系

建议将“用户-寰点关系”拆成三部分：

1. `ExplicitAffinity`
   由发文、选寰点、点赞、收藏、屏蔽、关注等显式行为计算
2. `VectorSimilarity`
   由用户向量与寰点向量的相似度计算
3. `FinalAffinity`
   由显式关系与隐式关系融合得到

示意公式：

```text
FinalAffinity(user, huandian) =
0.6 * ExplicitAffinity +
0.4 * VectorSimilarity
```

同理，也可以定义：

- `ArticleHuandianScore(article, huandian)`
- `UserArticleScore(user, article)`

这些融合分将用于：

- 候选寰点推荐
- 首页推荐排序
- 相似文章发现
- 潜在兴趣寰点发现

## 9. 第一版实施建议

如果按 MVP 推进，建议优先实现以下部分：

1. `huandians`
2. `post_huandians`
3. `user_huandian_affinities`
4. `huandian_votes`
5. `huandian_daily_stats`
6. `content_embeddings`

第一版建议做“轻量混合语义模型”，而不是纯规则系统：

- 寰点创建
- 寰点池展示
- 寰点认可投票
- 寰点回收判断
- 用户寰点粘性积累
- 文章向量生成
- 基于“粘性 + 向量相似度”的内容推荐

第二版再补充：

1. 寰点向量和用户向量
2. 自动归并建议
3. 更复杂的语义相似度
4. 域与寰点的自动流动关系
5. 主寰点与别名寰点的更精细治理
